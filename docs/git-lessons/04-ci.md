# Lesson 4 — CI: Let the Robot Check First

**Catch it before the reviewer sees it.**

---

## The scenario

The repo has a GitHub Actions workflow that runs on every PR. It runs `pytest` and a linter. If they fail, the PR can't be merged.

You push a change. CI goes red. You fix it. CI goes green. Reviewer never had to say a word about it.

---

## The workflow file

`.github/workflows/ci.yml` already exists on `main`. It looks like this:

```yaml
name: CI

on:
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      - run: pip install -e ".[dev]"
      - run: pytest
      - run: ruff check .
```

Every PR triggers this. If any step fails, GitHub blocks the merge.

---

## Make CI fail (on purpose)

```bash
git checkout -b feat/add-rename-command

# Add a new command but introduce a bug
# e.g. reference a variable that doesn't exist

git add .
git commit -m "wip: add rename command"
git push origin feat/add-rename-command

gh pr create --title "feat: rename command" --draft
```

Watch the CI check appear on the PR. Click the red X to see exactly which test failed and why.

---

## Fix it

```bash
# Fix the bug locally
git add tally/cli.py
git commit -m "fix rename command variable reference"
git push origin feat/add-rename-command
```

CI reruns automatically. Watch it go green.

---

## Key insight

> CI is a reviewer that never gets tired, never misses the same bug twice, and reviews at 3am. Branches make CI possible — you can't run checks on changes that don't exist yet as a separate unit.

---

## Branch protection (set this up on your fork)

On GitHub → Settings → Branches → Add rule for `main`:
- ✅ Require status checks to pass
- ✅ Require branches to be up to date

Now `main` physically cannot receive broken code.
